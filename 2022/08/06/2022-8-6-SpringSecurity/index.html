Spring Security 功能的主要实现主要是由 一系列过滤器链相互配合完成的

下面是部分过滤器

FilterSecurityInterceptor

这个过滤器是一个方法级别的权限过滤器,基本位于过滤器的最底部,用于保护 web 资源,使用 AccessDecisionManager 对当前用户进行授权访问

super.beforeInvocation(filterInvocation)

filterInvocation.getChain().doFilter(filterInvocation.getRequest(), filterInvocation.getResponse())

ExceptionTranslationFilter⏳

这个过滤器是一个异常过滤器,用来处理在认证授权过程中抛出的异常,能够捕获来及 FilterChain 所有异常并进行处理

但是它只会抛出两类异常 AuthenticationException 和 AccessDeniedException ,其它的异常会继续抛出

UsernamePasswordAuthenticationFilter

这个过滤器是对 /login 的 POST 请求做拦截，校验表单中用户名和密码。用于处理来自表单提交的认证。

该表单必须提供对应的用户名和密码，其内部还有登录成功或失败后进行处理的AuthenticationSuccessHandler 和 AuthenticationFailureHandler

两个接口

当我们在项目中什么也没有配置的时候，账号和密码是由 Spring Security 自动生成的。而在实际项目开发中，账号和密码都应该是从数据库中查询出来而不是由它帮我们生成。所以我们需要通过自定义逻辑控制认证逻辑

如果我们要自定义认证逻辑，我们需要编写一个类，让这个类继承 UsernamePasswordAuthenticationFilter 过滤器，在这个过滤器中，有一个 attemptAuthentication() 方法，这个方法用于接收 POST 请求过来的参数，然后实现认证逻辑。而认证的结果只有两种结果：要么成功，要么失败。而成功和失败的逻辑怎么处理，可以根据这个过滤器父类 AbstractAuthenticationProcessingFilter 中的 successfulAuthentication() 方法和unsuccessfulAuthentication() 方法来进行处理。如果认证成功，会进入 successfulAuthentication() 方法；如果认证失败，会进入unsuccessfulAuthentication() 方法。我们只需要重写这两个方法即可

简单说，就是编写一个类，继承 UsernamePasswordAuthenticationFilter 类，然后重写attemptAuthentication() 、successfulAuthentication() 和unsuccessfulAuthentication() 这三个方法。而查询数据库的方法需要写在 UserDetailsService 这个类中的 loadUserByUsername() 方法中

CSRF

概念

跨站请求伪造（英语：Cross-site request forgery），也被称为 one-click attack 或者 session riding，通常缩写为 CSRF 或者 XSRF， 是一种挟制用户在当前已登录的 Web 应用程序上执行非本意的操作的攻击方法。跟跨网站脚本（XSS）相比，XSS 利用的是用户对指定网站的信任， CSRF 利用的是网站对用户网页浏览器的信任

跨站请求攻击，简单地说，是攻击者通过一些技术手段欺骗用户的浏览器去访问一个自己曾经认证过的网站并运行一些操作（如发邮件，发消息，甚至财产操作如转账和购买 商品）。由于浏览器曾经认证过，所以被访问的网站会认为是真正的用户操作而去运行。 这利用了 web 中用户身份验证的一个漏洞：简单的身份验证只能保证请求发自某个用户的浏览器，却不能保证请求本身是用户自愿发出的。

从 Spring Security 4.0 开始，默认情况下会启用 CSRF 保护，以防止 CSRF 攻击应用程序，Spring Security CSRF 会针对 PATCH，POST，PUT 和 DELETE 方法进行防护
